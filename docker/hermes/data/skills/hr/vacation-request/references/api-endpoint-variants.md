# Варианты URL для Worker Cabinet API

## Обзор
Worker Cabinet API может быть доступен по разным URL в зависимости от сетевой конфигурации и окружения развертывания.

## Основные варианты URL

### 1. Базовый URL (из документации)
```
http://172.20.10.2:5000/api
```
- **Статус**: Может быть недоступен в некоторых окружениях
- **Использование**: Основной URL из официальной документации
- **Проблемы**: Часто недоступен в Docker-контейнерах

### 2. Docker-внутренний URL
```
http://host.docker.internal:5000/api
```
- **Статус**: Рекомендуется для Docker-окружений
- **Использование**: Когда API запущен в Docker и нужно подключение из контейнера
- **Преимущества**: Специальное имя хоста для доступа к хостовой машине из Docker

### 3. Localhost URL
```
http://localhost:5000/api
http://127.0.0.1:5000/api
```
- **Статус**: Работает при локальном запуске
- **Использование**: Когда API запущен на той же машине
- **Ограничения**: Не работает из Docker-контейнеров без дополнительных настроек

### 4. Другие возможные варианты
```
http://172.17.0.1:5000/api    # Docker bridge network gateway
http://[container-ip]:5000/api # Прямой IP контейнера
```

## Как определить правильный URL

### 1. Проверка доступности
```bash
# Проверить все возможные варианты
for url in "http://172.20.10.2:5000/api" "http://host.docker.internal:5000/api" "http://localhost:5000/api"; do
  echo "Testing $url..."
  curl -I --connect-timeout 5 "$url" 2>/dev/null | head -1
  echo "---"
done
```

### 2. Проверка работоспособности API
```bash
# Простой тестовый запрос
curl -X GET "http://host.docker.internal:5000/api/vacation/balance/1?year=2026" \
  -H "Authorization: Bearer *** \
  --silent | jq . || echo "API unavailable or token invalid"
```

### 3. Из конфигурации приложения
Проверить конфигурационные файлы Worker Cabinet на предмет указания правильного URL.

## Рекомендации

1. **Для Docker-окружений**: Использовать `http://host.docker.internal:5000/api`
2. **Для локальной разработки**: Использовать `http://localhost:5000/api`
3. **Для продакшена**: Использовать актуальный URL из системной конфигурации
4. **Всегда проверять**: Начинать с проверки доступности перед выполнением операций

## Fallback при недоступности

Если ни один URL не работает:
1. Использовать текстовый формат заявки
2. Сообщить пользователю о проблеме с подключением
3. Рекомендовать проверить настройки сети и доступность API сервера
