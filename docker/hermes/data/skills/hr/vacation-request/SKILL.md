# Создание заявки на отпуск

## Назначение
Помочь сотруднику оформить заявку на отпуск. Запросить недостающие данные, валидировать, оформить заявку.

## Данные пользователя

ФИО и должность берутся из профиля пользователя (USER PROFILE). ФИО **не спрашивать** — заявка всегда от имени того, кто пишет запрос.

## Требуемые данные от сотрудника

| Поле | Обязательно | Пример |
|------|-------------|--------|
| ФИО | Нет (из профиля) | — |
| Тип отпуска | Нет (по умолчанию «Ежегодный оплачиваемый») | Ежегодный оплачиваемый |
| За какой год (для ежегодного) | Нет (если не указан - текущий год) | 2025 |
| Дата начала | Да | 2025-07-01 |
| Дата окончания | Да | 2025-07-14 |
| Количество дней | Нет (вычисляется) | 14 |
| Примечание | Нет | Отпуск за свой счет |

## Допустимые типы отпуска

- Ежегодный оплачиваемый
- Отпуск без сохранения заработной платы
- Больничный
- Учебный
- Декретный
- Отпуск за работу во вредных условиях

## Шаги

1. Запросить даты отпуска (начало и окончание).
2. Если тип отпуска не указан — считать «Ежегодный оплачиваемый».
3. Если тип — ежегодный оплачиваемый — спросить, за какой год отпуск (если не указан, использовать текущий год).
4. Запросить остальные недостающие данные (см. таблицу выше).
5. Валидация:
   - Дата окончания >= даты начала
   - Даты в будущем
   - Если количество дней не указано — вычислить как (окончание - начало + 1)
   - ФИО не пустое
3. Оформить заявку в формате, указанном ниже.
4. Подтвердить сотруднику, что заявка создана.
}
```

## Формат заявки

```
ЗАЯВКА НА ОТПУСК

Сотрудник: {ФИО}
Должность: {Должность, если известна}
Тип отпуска: {Тип}
Период: {дата начала} — {дата окончания}
Количество дней: {кол-во}
Примечание: {примечание или «—»}

Дата создания: {текущая дата}
```

## Проверки

- Если сотрудник не указал какой-либо обязательный параметр — попросить уточнить, НЕ подставлять значения (кроме типа отпуска — он по умолчанию «Ежегодный оплачиваемый», и года — текущий год по умолчанию).
- Если даты некорректны — указать ошибку и попросить исправить.

## Интеграция с Worker Cabinet API

Скилл интегрирован с API Worker Cabinet. 

**ВАЖНО**: URL API может различаться в зависимости от окружения:
- `http://172.20.10.2:5000` - базовый URL (может быть недоступен)
- `http://host.docker.internal:5000` - URL для Docker-окружения
- Другие URL в зависимости от сетевой конфигурации

Всегда проверяйте доступность API перед использованием.

**Дополнительные материалы**:
- Смотрите `references/api-connection-troubleshooting.md` для диагностики проблем с подключением к API и fallback-процедур
- Смотрите `references/api-endpoint-variants.md` для различных вариантов URL API
- Используйте `templates/worker-cabinet-client-template.js` как основу для Node.js клиента
- Используйте `scripts/test_worker_cabinet_api.sh` для тестирования доступности API

### Требования
- Наличие Bearer токена для авторизации
- URL API: `http://172.20.10.2:5000`

### Основные эндпоинты

#### 1. Создание заявки на отпуск
```bash
curl -X POST http://172.20.10.2:5000/vacation/requests \
  -H "Authorization: Bearer YOUR_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "startDate": "2025-07-01",
    "endDate": "2025-07-14", 
    "vacationType": "annual_paid",
    "comment": "Ежегодный отпуск",
    "hasTravel": false
  }'
```

#### 2. Получение баланса отпусков
```bash
curl -X GET "http://172.20.10.2:5000/vacation/balance/1?year=2025" \
  -H "Authorization: Bearer YOUR_TOKEN"
```

#### 3. Проверка ограничений перед созданием заявки
```bash
curl -X POST http://172.20.10.2:5000/vacation/check-restrictions \
  -H "Authorization: Bearer YOUR_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "userId": 1,
    "startDate": "2025-07-01",
    "endDate": "2025-07-14"
  }'
```

#### 4. Получение списка заявок
```bash
curl -X GET "http://172.20.10.2:5000/vacation/requests?userId=1&status=pending&year=2025" \
  -H "Authorization: Bearer YOUR_TOKEN"
```

#### 5. Получение типов отпусков
```bash
curl -X GET http://172.20.10.2:5000/dictionaries/vacation-types \
  -H "Authorization: Bearer YOUR_TOKEN"
```

#### 6. Генерация заявления на отпуск (DOCX)
```bash
curl -X POST http://172.20.10.2:5000/vacation/generate-application \
  -H "Authorization: Bearer YOUR_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "year": 2025,
    "templateId": 1
  }' \
  --output vacation-application.docx
```

### Типы отпусков (vacationType)
- `annual_paid` - Ежегодный оплачиваемый
- `unpaid` - Без сохранения заработной платы  
- `educational` - Учебный
- `maternity` - Декретный
- `child_care` - По уходу за ребенком
- `additional` - Дополнительный
- `veteran` - Ветеранский

### Статусы заявок (status)
- `pending` - На рассмотрении
- `approved` - Одобрено
- `rejected` - Отклонено
- `cancelled` - Отменено

### Алгоритм работы с API

1. **Определение правильного URL**: Проверить доступность API по разным адресам
2. **Авторизация**: Использовать Bearer токен в заголовке `Authorization`
3. **Проверка доступности API**: Проверить, что сервер API доступен
4. **Проверка баланса**: Получить текущий баланс отпусков для пользователя
5. **Проверка ограничений**: Проверить, нет ли ограничений на выбранные даты
6. **Создание заявки**: Отправить запрос на создание заявки с валидными данными
7. **Обработка ответа**: Проверить статус ответа и сообщить пользователю результат

#### Проверка доступности API
```bash
# Быстрая проверка нескольких URL
urls=("http://172.20.10.2:5000/api" "http://host.docker.internal:5000/api")
for url in "${urls[@]}"; do
  if curl -I --connect-timeout 3 "$url" >/dev/null 2>&1; then
    API_URL="$url"
    break
  fi
done
```

#### Рекомендуемый метод: Node.js клиент

Для надежной работы с API предпочтительнее использовать Node.js клиент вместо прямых curl команд:

```javascript
const WorkerCabinetClient = require('./index');
const client = new WorkerCabinetClient(API_URL, TOKEN);

const vacationData = {
  startDate: '2026-11-02',
  endDate: '2026-11-20',
  vacationType: 'annual_paid',
  comment: 'Отпуск за 2026 год',
  hasTravel: false
};

try {
  const request = await client.createVacationRequest(vacationData);
  console.log('Заявка создана:', request.id);
} catch (error) {
  console.error('Ошибка:', error.message);
}
```

#### Альтернатива: Curl с JSON-файлом

При проблемах с экранированием в curl использовать отдельный JSON-файл:

```bash
# 1. Создать JSON файл
cat > vacation_data.json << EOF
{
  "startDate": "2026-11-02",
  "endDate": "2026-11-20",
  "vacationType": "annual_paid",
  "comment": "Отпуск за 2026 год",
  "hasTravel": false
}
EOF

# 2. Использовать файл в запросе
curl -X POST "$API_URL/vacation/requests" \
  -H "Authorization: Bearer *** \
  -H "Content-Type: application/json" \
  -d @vacation_data.json
```

### Обработка ошибок
- `400 VALIDATION_ERROR` - Ошибка валидации данных
- `401 UNAUTHORIZED` - Неверный токен авторизации
- `403 FORBIDDEN` - Недостаточно прав
- `404 NOT_FOUND` - Ресурс не найден или эндпоинт недоступен
- `409 CONFLICT` - Конфликт (например, пересечение с другим отпуском)

### ПИТФАЛ: Проблемы с токеном авторизации

**Проблема**: При использовании curl команд возникают ошибки `"Invalid or expired token"` даже при корректном токене.

**Причины**:
- Токен истек или был отозван
- Проблемы с экранированием кавычек в shell-скриптах
- Несоответствие формата токена
- Проблемы с сетевой доступностью сервера

**Диагностика**:
```bash
# Проверить баланс отпусков (простая операция)
curl -X GET "http://host.docker.internal:5000/api/vacation/balance/1?year=2026" \
  -H "Authorization: Bearer YOUR_TOKEN"
```

**Решение**:
1. Использовать Node.js клиент для надежной работы с API
2. Проверять токен через простые GET-запросы перед POST
3. При проблемах с curl использовать JSON-файлы для данных:
```bash
# Создать JSON файл
echo '{
  "startDate": "2026-11-02",
  "endDate": "2026-11-20",
  "vacationType": "annual_paid",
  "comment": "Отпуск за 2026 год",
  "hasTravel": false
}' > vacation_data.json

# Использовать файл в запросе
curl -X POST "http://host.docker.internal:5000/api/vacation/requests" \
  -H "Authorization: Bearer YOUR_TOKEN" \
  -H "Content-Type: application/json" \
  -d @vacation_data.json
```

### ПИТФАЛ: Ограничения по отделу

**Проблема**: API может вернуть ошибку `"Превышен лимит одновременных отпусков в отделе (макс. 1)"`.

**Причина**: В отделе уже есть сотрудник в отпуске на выбранные даты, и по политике компании разрешено не более 1 одновременного отпуска.

**Решение**:
1. Сообщить пользователю о наличии ограничения
2. Рекомендовать согласовать даты с руководителем
3. Предложить выбрать другие даты
4. При API-ошибке fallback в текстовый формат с пометкой о необходимости согласования

### ПИТФАЛ: Недоступность API сервера

**Проблема**: Сервер Worker Cabinet API может быть недоступен по адресу `http://172.20.10.2:5000`

**Причины**:
- Сервер не запущен
- Сервер работает на другом IP/порту
- Сетевые проблемы или防火墙 блокирует доступ
- Неверный URL в конфигурации

**Диагностика**:
```bash
# Проверка доступности сервера
curl -I http://172.20.10.2:5000
```

**Решение / Fallback**:
1. Если API недоступен, подготовить заявку в текстовом формате
2. Использовать стандартный формат заявки (см. раздел "Формат заявки")
3. Сообщить пользователю о недоступности API и необходимости подачи заявки через другие каналы
4. Рекомендовать проверить доступность API сервера у системного администратора

**Пример fallback-ответа**:
```
📋 ЗАЯВКА НА ОТПУСК

Сотрудник: {ФИО}
Должность: {Должность}
Тип отпуска: {Тип}
Период: {дата начала} — {дата окончания}
Количество дней: {кол-во}
Примечание: {примечание или «—»}

Дата создания: {текущая дата}
```

**Примечание**: Всегда сначала проверять доступность API перед выполнением других операций. При ошибках подключения немедленно переходить в fallback-режим.

### ПИТФАЛ: Блокировка заявок HR-отделом

**Проблема**: API может вернуть ошибку `"Подача заявок на отпуск для вашего отдела временно заблокирована HR"` или подобную, указывающую на административные ограничения.

**Причины**:
- HR-отдел ввел временные ограничения на подачу заявок для определенных отделов
- Превышен лимит одновременных отпусков в отделе
- Период закрытия отпусков или другие организационные ограничения
- Необходимость предварительного согласования с руководством

**Диагностика**: API возвращает ошибку с сообщением о блокировке HR-отдела.

**Решение / Fallback**:
1. При получении ошибки о блокировке немедленно переходить в текстовый формат заявки
2. Добавить явное указание о блокировке в примечание к заявке
3. Рекомендовать пользователю следующие действия:
   - Связаться с HR-отделом для выяснения причин блокировки
   - Согласовать отпуск с непосредственным руководителем
   - Уточнить возможные альтернативные даты
4. В текстовой заявке указать, что требуется согласование

**Пример fallback-ответа при блокировке HR**:
```
📋 ЗАЯВКА НА ОТПУСК

Сотрудник: {ФИО}
Должность: {Должность}
Тип отпуска: {Тип}
Период: {дата начала} — {дата окончания}
Количество дней: {кол-во}
Примечание: —

Дата создания: {текущая дата}

Примечание: Подача заявок на отпуск для вашего отдела временно заблокирована HR. Пожалуйста, согласуйте заявку с руководителем или HR-отделом.
```

**Алгоритм обработки**:
1. Попытка создания заявки через API
2. При ошибке блокировки HR:
   - Определить тип ошибки (department block, limit exceeded, etc.)
   - Сформировать текстовую заявку с соответствующим примечанием
   - Предоставить четкие инструкции пользователю по дальнейшим действиям
3. Сохранить копию заявки для возможной последующей подачи через другие каналы
