# Система авторизации: Отчет о проверке

## Обзор
Система авторизации worker-cabinet основана на JWT (JSON Web Tokens) и ролевой модели доступа (RBAC).

## Архитектура

### Компоненты

1. **Backend:**
   - `backend/src/middleware/auth.js` - Middleware для аутентификации и авторизации
   - `backend/src/routes/auth.js` - API endpoints для авторизации
   - `backend/src/routes/users.js` - API endpoints для пользователей с RBAC
   - `backend/src/routes/vacation.js` - API endpoints для заявок с RBAC

2. **Frontend:**
   - `store/authStore.ts` - Zustand store для управления состоянием авторизации

### Поток авторизации

1. **Регистрация:**
   - POST `/api/auth/register`
   - Пароль хешируется bcrypt (10 rounds)
   - Создается JWT токен
   - Создается баланс отпускных дней

2. **Вход:**
   - POST `/api/auth/login`
   - Проверка email и пароля
   - Генерация JWT токена
   - Возврат токена и данных пользователя

3. **Проверка токена:**
   - Middleware `authenticateToken` проверяет заголовок Authorization
   - JWT токен верифицируется с секретным ключом
   - Пользователь добавляется в `req.user`

4. **Ролевая проверка:**
   - Middleware `authorizeRoles` проверяет роль пользователя
   - Поддерживаемые роли: employee, manager, hr, admin

## Роли и права доступа

| Роль | Права |
|------|-------|
| **employee** | Доступ только к своим данным, создание заявок, отмена своих заявок |
| **manager** | Доступ к своим данным + подчиненным, одобрение/отклонение заявок |
| **hr** | Доступ ко всем данным и заявкам, полные права |
| **admin** | Все права HR + административные функции |

## Безопасность

### Сильные стороны ✓

1. **Хеширование паролей:**
   - Используется bcrypt с 10 rounds
   - Пароли никогда не хранятся в открытом виде

2. **JWT токены:**
   - Токены имеют срок действия (7 дней по умолчанию)
   - Содержат только необходимые данные (id, email, role)

3. **Ролевая модель:**
   - Четкое разделение прав доступа
   - Middleware для проверки ролей на backend

4. **Защита маршрутов:**
   - Все защищенные маршруты требуют токен
   - Проверка прав доступа на уровне API

5. **Проверка прав на уровне данных:**
   - Пользователи видят только свои данные
   - Менеджеры видят данные своих подчиненных

### Потенциальные улучшения ⚠️

1. **JWT секрет:**
   - В .env: `JWT_SECRET=your-secret-key-change-in-production`
   - **Рекомендация:** Использовать сильный случайный секрет в production

2. **Срок действия токена:**
   - 7 дней может быть слишком долго для критичных приложений
   - **Рекомендация:** Рассмотреть сокращение до 24-48 часов

3. **Refresh токены:**
   - Не реализованы
   - **Рекомендация:** Добавить refresh токены для лучшего UX

4. **Rate limiting:**
   - Не реализован
   - **Рекомендация:** Добавить rate limiting для auth endpoints

5. **Логирование попыток входа:**
   - Базовое логирование
   - **Рекомендация:** Добавить детальное логирование попыток входа

6. **Блокировка аккаунта:**

## Анализ кода

### auth.js:3-18 - authenticateToken
```javascript
export const authenticateToken = (req, res, next) => {
  const authHeader = req.headers['authorization']
  const token = authHeader && authHeader.split(' ')[1]

  if (!token) {
    return res.status(401).json({ error: 'Access token required' })
  }

  jwt.verify(token, process.env.JWT_SECRET, (err, user) => {
    if (err) {
      return res.status(403).json({ error: 'Invalid or expired token' })
    }
    req.user = user
    next()
  })
}
```
✓ Правильно реализует middleware pattern
✓ Правильно обрабатывает ошибки
✓ Добавляет пользователя в request

### auth.js:20-32 - authorizeRoles
```javascript
export const authorizeRoles = (...roles) => {
  return (req, res, next) => {
    if (!req.user) {
      return res.status(401).json({ error: 'Unauthorized' })
    }

    if (!roles.includes(req.user.role)) {
      return res.status(403).json({ error: 'Forbidden: Insufficient permissions' })
    }

    next()
  }
}
```
✓ Использует замыкание для хранения ролей
✓ Правильно проверяет доступ
✓ Возвращает корректные HTTP статусы

## Рекомендации по улучшению

### Критические (High Priority)

1. **Изменить JWT_SECRET в production**
   - Использовать: `openssl rand -base64 64`
   - Добавить в переменные окружения

2. **Добавить HTTPS в production**
   - Все API endpoints должны быть доступны только по HTTPS
   - Настроить HTTP/2

### Важные (Medium Priority)

3. **Добавить rate limiting**
   - Для auth endpoints: 5 попыток в минуту
   - Использовать express-rate-limit

4. **Сократить срок действия токена**
   - С 7 дней до 24-48 часов
   - Добавить refresh токены

5. **Добавить валидацию ввода**
   - Использовать Joi (уже в зависимостях)
   - Валидировать email, password, и т.д.

### Желательные (Low Priority)

6. **Добавить логирование попыток входа**
   - Успешные и неудачные попытки
   - IP адреса и timestamps

7. **Добавить 2FA**
   - Для HR и admin ролей
   - Опционально для всех пользователей

8. **Добавить password recovery**
   - Функция сброса пароля
   - Email подтверждение
